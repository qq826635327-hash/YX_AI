## 任务：修复 Worker 心跳误报 + 诊断逻辑纠偏（用户真实数据触发）

### 触发
用户贴了一份真实 session 诊断报告：FPS 全程 31-32（127/127 采样 <50）、block 70 次全来自 worker、longtask=0。我的诊断给出 critical"主线程阻塞1014ms"指向找阻塞脚本——**错误方向**。

### 根因
我的 Worker 心跳设计有缺陷：
1. WORKER_BLOCK_THRESHOLD_MS=300ms 太低，与 rAF 的 200ms 阈值重叠
2. Worker 内部有 setInterval 主动检测（lastPong 超时即报 block），主线程持续繁忙（30fps 每帧满载）时 Worker 收 ping 被延迟 → 误报"阻塞"
3. 但实际上 rAF 仍在触发（delta <200ms），主线程没冻结，只是每帧渲染过重
4. diagnose_session 无条件把 block>200ms 判为 critical，没区分来源(raf vs worker)和 FPS 状态

### 修复
**observer.ts**：
- WORKER_BLOCK_THRESHOLD_MS: 300→1500（必须远大于 rAF 200ms，只在真正冻结时触发）
- Worker 代码：删掉内部 setInterval 主动检测，只回 pong（判定逻辑全归主线程）
- 主线程 workerCheckTimer：增加 `rafStalled` 双重确认——只有 `rAF 停了 AND Worker 超时` 才报 worker block；单纯 Worker 慢但 rAF 在跑 = 繁忙不算冻结
- ping 间隔 100→200ms 减少开销

**perf_service.py diagnose_session**：
- block 部分：检查 block 事件来源。全 worker + FPS 仍活跃 → 降级 warning"主线程繁忙（非冻结）"；含 raf 来源 → 才是 critical"主线程冻结"
- FPS 部分：新增场景识别——FPS 低 + longtask=0 + 无真冻结 → "每帧渲染过重"（温水煮青蛙式卡顿），给出 5 条针对性排查方向（虚拟列表/React Profiler/动画属性/state 频率/每帧计算）

### 验证
- 对同一 session 重新诊断：severity 从 critical 降为 warning
- finding1: "Worker 心跳70次超时但rAF仍在触发" warning（说明非冻结）
- finding2: "持续低帧率31fps但无长任务，疑似每帧渲染过重" warning（正确方向）
- tsc 0 错误、build 成功、后端重启健康检查 OK

### 经验教训
- Worker 心跳兜底只能用于"主线程彻底冻结"场景（rAF 完全停止），不能用于"繁忙"场景
- 阈值设计要考虑重叠区间：rAF 200ms 和 Worker 300ms 重叠导致两者竞争上报，冷却逻辑让 Worker 抢到了窗口
- 诊断必须交叉验证多个指标：block 来源 + FPS 是否活跃 + longtask 数量，单看一个指标会误判
- "持续低FPS无长任务"是最容易被忽略的卡顿类型——每帧都超支但无单点 >50ms，longtask observer 抓不到
