# PRD-01 · 产品概览与愿景

> **版本**：v1.0 · **Owner**：产品负责人 

## 1. 产品身份

- **产品名称**：影序 AI Drama Studio（内部代号 ADS）
- **产品形态**：本地化部署的 Web 工作台（FastAPI + React SPA）
- **当前版本**：v0.1.0（Phase 3 进行中）
- **目标用户**：AI 短剧 / 漫剧独立创作者（"一人剧组"）
- **产品口号**：*从一行剧本到一段短剧，全在本地完成。*

## 2. 产品愿景

> 让一个没有任何美术资源的创作者，在自家电脑上，用一套工具完成"编剧 → 角色设计 → 场景绘制 → 分镜排版 → 视频生成"的全流程短剧生产。

## 3. 产品定位与边界

### 3.1 做什么

- 提供 **剧本 → AI 解析 → 实体资产 → AI 生成图片 / 视频** 的结构化生产流水线
- 管理 **项目级** 的剧本、角色、场景、道具、剧集、分镜、资产
- 对接 **多 Provider** 的 AI 模型（Agnes 已接入，SenseNova / OpenAI-compatible 预留，ComfyUI 规划中）
- 提供 **任务中心** 实时跟踪所有 AI 生成任务
- 提供 **本地化优先** 的数据存储（SQLite + 本地文件系统 + Fernet 加密 API Key）

### 3.2 不做什么（Scope Exclusion）

| 明确不做         | 原因           |
| ------------ | ------------ |
| 移动端 App      | 工作台形态决定桌面优先  |

## 4. 核心价值主张（Value Proposition）

| 用户痛点                                     | 影序 AI 的解法                            |
| ---------------------------------------- | ------------------------------------ |
| 短剧生产工具散落（PS + 剪映 + ChatGPT + 各种 AI 绘画平台） | 一站式工作台，所有资产与 Prompt 集中在一个项目树里        |
| AI 生成结果无法追溯                              | 每次生成任务都有结构化日志 + trace_id + 关联实体      |
| 多模型 API Key 管理混乱                         | Fernet 加密本地存储，Provider → Model 两层结构  |
| 本地文件与数据库状态不一致                            | 双向资产同步（DB ↔ 磁盘），一键修复                 |
| 剧本与分镜割裂                                  | 7 段式 Prompt 结构 + 剧本 AI 解析流水线，保持上下文一致 |

## 5. 成功指标（North Star + Driver Metrics）

### 5.1 North Star Metric

**周活跃创作者数（WAC）**：7 天内至少完成 1 次 AI 解析 + 1 次 AI 生成的独立项目数。

### 5.2 Driver Metrics

| 指标 | 定义 | 当前基线 | Phase 3 目标 |
|---|---|---|---|
| 单项目从剧本到首支视频的端到端耗时 | 从 `parse_status = parsed` 到第一个 `shot_video` 资产生成 | N/A（Phase 2 无视频） | < 2 小时 |
| AI 解析成功率 | `parse_status = parsed` 的项目占比 | ~85% | > 95% |
| AI 生成任务成功率 | `status = succeeded` 的生成任务占比 | N/A | > 80% |
| 单次会话时长 | 用户打开应用到关闭的时长 | N/A | > 30 分钟 |
| 任务失败后重试率 | `failed` → `retry` 的比例 | N/A | > 40%（表明用户仍信任系统） |

### 5.3 Counter Metrics（防止过度优化）

- 单用户单日失败任务 > 10 次占比（健康度反向指标）
- 单项目磁盘占用 > 5 GB 的项目占比（存储健康度）
- SQLite WAL 文件残留事件数（稳定性反向指标）

## 6. 产品原则（Product Principles）

1. **本地优先（Local First）**：所有数据本地存储，API Key 加密，不上传云端。
2. **结构化驱动（Structured by Default）**：剧本 → 实体 → 分镜 → 资产，每一步都结构化，AI 可读、人可编辑。
3. **任务可追溯（Traceable Tasks）**：每次 AI 生成都可追溯到 Provider / Model / Prompt / Reference / TraceID。
4. **渐进式能力（Progressive Capability）**：新手看到简洁界面，高级用户可配置 Provider、ComfyUI、提示词模板。
5. **AI 友好（AI-Friendly）**：日志结构化、WebSocket 实时推送、API 契约稳定，便于 AI 助手参与开发。

## 11. UI 架构与整体布局

### 11.1 BR-UI-01：三区布局架构

**需求描述**：应用采用固定三区布局（TopBar + Sidebar + Content），所有页面共享统一的视觉骨架。

```
┌────────────────────────────────────────────────────────────┐
│ TopBar（顶栏：当前项目名、模块切换按钮、用户操作菜单）         │
├──────────────┬─────────────────────────────────────────────┤
│              │                                             │
│ Sidebar      │  Content（主内容区）                          │
│（左侧导航栏） │  每个页面使用统一 PageContainer 布局           │
│              │                                             │
│ 根据当前模块  │                                             │
│ 动态展示菜单  │                                             │
└──────────────┴─────────────────────────────────────────────┘
```

**功能需求**：

| FR 编号 | 功能项 | 描述 |
|---|---|---|
| FR-UI-01 | TopBar 固定栏 | 固定在顶部，显示当前项目名、模块切换（项目/任务/配置）、用户操作菜单 |
| FR-UI-02 | Sidebar 动态菜单 | 左侧导航栏，根据当前模块（项目工作台/任务中心/配置中心）动态切换菜单项 |
| FR-UI-03 | Content 主内容区 | 右侧主区域，承载所有业务页面，统一 PageContainer 布局 |
| FR-UI-04 | 模块切换 | 通过 TopBar 按钮在「项目」「任务」「配置」三大模块间切换 |

**验收标准**：

- AC-UI-01：进入项目工作台后，Sidebar 显示「剧本 / 角色 / 场景 / 道具 / 剧集结构」五项
- AC-UI-02：切换到任务中心后，Sidebar 显示「全部 / 进行中 / 已完成 / 失败 / 已取消」五项过滤
- AC-UI-03：切换到配置中心后，Sidebar 显示「API 供应商 / 插件扩展 / 模型配置 / 提示词模版 / ComfyUI 服务器 / ComfyUI 工作流」
- AC-UI-04：所有页面共享统一的 PageContainer 布局，无页面级自定义外壳

### 11.2 用户故事

- **US-UI-01**：作为创作者，我希望应用有统一的导航结构，这样我不用记住每个功能的入口路径。
- **US-UI-02**：作为创作者，我希望 Sidebar 根据我当前所在模块自动切换菜单，这样界面不会过于拥挤。

## 12. 实时反馈与通信

### 12.1 BR-RT-01：WebSocket 实时推送

**需求描述**：所有异步操作（AI 生成任务、剧本解析）的进度通过 WebSocket 实时推送到前端，用户无需手动刷新。

| FR 编号 | 功能项 | 描述 |
|---|---|---|
| FR-RT-01 | 任务进度推送 | AI 生成任务的创建、进度、完成、失败通过 `/ws/tasks` 实时推送 |
| FR-RT-02 | 剧本解析推送 | 三阶段解析管线（提取 → 结构化 → 校验）的进度通过 `/ws/script` 实时推送 |
| FR-RT-03 | 日志推送 | 系统日志和任务结构化日志通过 `/ws/logs` 实时推送到前端 LogViewer |
| FR-RT-04 | 轮询降级 | WebSocket 断开时自动启用 5 秒轮询作为降级方案，重新连接后自动停止轮询 |
| FR-RT-05 | 重连策略 | 采用指数退避 + ±25% 随机 jitter，避免多客户端同时重连造成惊群效应 |

**验收标准**：

- AC-RT-01：用户提交 AI 生成后，任务卡片在 1 秒内更新状态，无需刷新页面
- AC-RT-02：WebSocket 断开后，前端在 5 秒内降级为轮询，用户无感知
- AC-RT-03：WebSocket 恢复后，轮询自动停止，不产生额外请求
- AC-RT-04：服务端重启后，所有客户端分散重连，不出现瞬时并发洪峰

### 12.2 BR-RT-02：通知与错误处理

| FR 编号 | 功能项 | 描述 |
|---|---|---|
| FR-RT-06 | Toast 通知 | 操作成功/失败通过右上角 Toast 提示，默认 3 秒消失，同类型不堆叠 |
| FR-RT-07 | ErrorBoundary | 异常时显示友好降级页，提供「刷新页面」和「复制错误」按钮，不会白屏 |
| FR-RT-08 | API 错误统一处理 | 所有 API 错误通过 `ApiError` 类统一抛出，包含错误类型和人类可读描述 |

### 12.3 BR-RT-03：自动保存

| FR 编号 | 功能项 | 描述 |
|---|---|---|
| FR-RT-09 | 即时保存 | 除剧本等大型编辑外，其他修改即时保存 |
| FR-RT-10 | debounce 保存 | 提示词输入框带 debounce 自动保存 |
| FR-RT-11 | 保存失败提示 | PATCH 失败时显示 Toast 错误 |

### 12.4 用户故事

- **US-RT-01**：作为创作者，我希望提交生成任务后能实时看到进度，这样我不需要反复刷新页面。
- **US-RT-02**：作为创作者，我希望网络不稳定时系统能自动降级，这样我不会因为断连丢失操作反馈。

## 14. 数据存储策略

### 14.1 BR-DATA-01：本地化存储架构

**需求描述**：所有用户数据在本地存储，不依赖云端服务，保障隐私和离线可用性。

| 数据类型 | 存储位置 | 加密 | 说明 |
|---|---|---|---|
| 业务数据 | 本地 SQLite（WAL 模式） | — | 单文件数据库 `data/db/app.sqlite`，支持并发读写 |
| 文件存储 | 项目目录下分类型目录 | — | `images/`、`videos/`、`prompts/` 按实体组织 |
| API Key | 数据库 `encrypted_api_key` 字段 | Fernet 对称加密 | 生产环境必须通过 `ADS_ENCRYPTION_KEY` 环境变量注入密钥 |
| 服务端状态 | React Query 缓存 | — | 自动管理缓存失效和重新获取 |
| 客户端 UI 状态 | Zustand stores | — | 侧边栏折叠、主题、Toast 等纯 UI 状态 |
| 主题偏好 | localStorage | — | 跟随浏览器持久化 |

**验收标准**：

- AC-DATA-01：完全断开网络时，应用所有 CRUD 功能正常可用
- AC-DATA-02：API Key 在数据库中以加密形式存储，直接查看 SQLite 无法看到明文
- AC-DATA-03：WAL 模式启用，支持读写并发（写不阻塞读）
- AC-DATA-04：异常退出后，WAL 文件可手动清理恢复（已知行为，见 Bug 记录 §14）

### 14.2 BR-DATA-02：数据备份（用户责任）

| FR 编号 | 功能项 | 描述 |
|---|---|---|
| FR-DATA-01 | 数据库位置 | 项目数据存于 `data/db/app.sqlite` |
| FR-DATA-02 | 文件位置 | 项目文件存于 `data/projects/` |
| FR-DATA-03 | 备份建议 | 建议用户定期备份整个 `data/` 目录 |
| FR-DATA-04 | 不做云备份 | 应用不做自动云备份，完全本地化 |

### 14.3 用户故事

- **US-DATA-01**：作为创作者，我希望所有数据都在本地，这样我的创意素材不会上传到任何云端。
- **US-DATA-02**：作为创作者，我希望 API Key 加密存储，这样即使数据库文件泄露也不会暴露密钥。

## 15. 状态管理架构

### 15.1 BR-STATE-01：分层状态管理

**需求描述**：前端状态按职责分层管理，服务端数据与 UI 状态严格分离。

| 状态类型 | 管理工具 | 示例 | 规则 |
|---|---|---|---|
| 服务端状态 | TanStack Query | 项目列表、角色列表、任务列表 | 自动缓存、失效、重新获取 |
| 客户端 UI 状态 | Zustand | 侧边栏折叠、主题、Toast | 纯前端状态，不涉及服务端 |
| 局部组件状态 | useState / useReducer | 表单输入、对话框开关 | 组件内部管理 |
| 派生状态 | useMemo | 过滤后的列表 | 从已有状态计算，不额外存储 |

**强制规则**：

- FR-STATE-01：服务端数据**禁止**存入 Zustand，必须通过 TanStack Query 管理
- FR-STATE-02：API 调用**必须**走 `frontend/src/api/` 层，禁止组件内直接 fetch
- FR-STATE-03：类型定义**统一**放 `frontend/src/types/index.ts`

### 15.2 用户故事

- **US-STATE-01**：作为开发者，我希望状态管理有清晰的职责划分，这样不会出现数据不同步的 Bug。
