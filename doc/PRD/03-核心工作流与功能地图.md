### 1.2 工作流关键数据流

| 起点 | 终点 | 数据类型 | 触发方式 |
|---|---|---|---|
| 剧本.raw_text | ScriptDocument.parsed_result | LLM 解析结果（JSON） | 用户点击"AI 解析" |
| parsed_result | Character / Scene / Prop / Episode | 实体记录 | script_parser 流水线 |
| Character / Scene / Prop | Asset (image) | AI 生成图 | 用户点击"生成" |
| Shot (7 段 Prompt) | Asset (first_frame / last_frame / video) | AI 生成媒体 | 用户点击"生成" |
| Shot.references | ShotCharacter / ShotScene / ShotProp | 引用关系 | 用户在分镜页手动关联 |
| Asset.file_path | 磁盘文件 | 双向同步 | 用户点击"同步"按钮 |

### 1.3 七步 Happy Path（端到端主流程）


| 步骤 | 操作 | 目标页面 | 关联 PRD |
|---|---|---|---|
| **Step 1** | 新建项目 | 首页项目列表 | PRD-10 |
| **Step 2** | 录入完整剧本 | 剧本编辑器（60vh 文本区） | PRD-10 |
| **Step 3** | AI 解析 → 自动生成角色 / 场景 / 道具 / 剧集 | 剧本页（三阶段管线 + WebSocket 进度） | PRD-10 |
| **Step 4** | 角色 / 场景 / 道具各自生成「主图」 | 角色页 / 场景页 / 道具页（GenerateDialog） | PRD-11, PRD-13 |
| **Step 5** | 进入剧集结构，逐分镜补全描述 | 剧集页（展开分镜网格 + 7 段式 Prompt 编辑） | PRD-12 |
| **Step 6** | 分镜级生成首帧 / 尾帧 / 视频 | 分镜卡片 → GenerateDialog | PRD-13 |
| **Step 7** | 在任务中心统一跟踪所有生成任务 | 任务中心（状态过滤 + 重试 + 取消） | PRD-14 |

**用户故事**：

- **US-HP-01**：作为首次使用的创作者，我希望按照 7 步指引就能产出第一支短剧片段，这样我不会在功能间迷路。
- **US-HP-02**：作为熟练创作者，我希望在 Step 4–6 之间自由跳转，不被线性流程约束。

**关键前置条件**：

- Step 3（AI 解析）需要先完成 Step 2（剧本录入），且至少配置一个可用的 AI Provider
- Step 4/6（AI 生成）依赖 Provider 的 API Key 已配置并通过连接测试
- Step 6（分镜生成）依赖 Step 3 的 AI 解析产出（Shot 记录）

## 6. M9 系统维护与可靠性 — 详细功能规格

> 本节记录已实现的系统级维护和可靠性增强功能。这些功能在后台自动运行，对用户透明。

### 6.1 FR-M9-01：数据库备份与 WAL Checkpoint

**需求描述**：应用启动时自动执行 SQLite WAL checkpoint 并创建数据库快照备份，防止数据丢失和 WAL 文件膨胀。

| 功能项 | 描述 |
|---|---|
| 启动时 WAL Checkpoint | 应用启动后自动执行 `PRAGMA wal_checkpoint(TRUNCATE)`，将 WAL 文件内容合并回主数据库 |
| 快照备份 | WAL checkpoint 成功后，复制 `app.sqlite` 为 `app.sqlite.bak`（保留最近 1 份） |
| 失败处理 | checkpoint 或备份失败时记录 WARNING 日志，不阻塞应用启动 |

**验收标准**：

- AC-M9-01：应用正常启动后，`data/db/app.sqlite-wal` 文件大小趋近于 0
- AC-M9-02：`data/db/app.sqlite.bak` 文件存在且时间戳为最近一次启动时间
- AC-M9-03：备份失败不影响应用正常启动

### 6.2 FR-M9-02：智能降级（Smart Degradation）

**需求描述**：AI API 调用失败时，自动尝试同类型的备用 Provider，减少用户手动干预。

| 功能项 | 描述 |
|---|---|
| 自动回退 | 主 Provider API 调用失败（超时 / 5xx / 连接错误）时，自动切换到备用 Provider |
| 降级日志 | 每次降级触发时记录结构化日志（原 Provider → 备用 Provider → 原因） |
| 降级上限 | 最多尝试 2 个 Provider（1 主 + 1 备），超过则标记任务为 failed |

**验收标准**：

- AC-M9-04：主 Provider 返回 5xx 时，任务自动切换到备用 Provider，前端显示降级提示
- AC-M9-05：两个 Provider 均失败时，任务状态为 `failed`，结构化日志包含两次调用详情
- AC-M9-06：降级事件通过 WebSocket 推送到前端，用户可在任务日志中看到

### 6.3 FR-M9-03：提示词缓存（Prompt Caching）

**需求描述**：相同参数的 AI 生成请求跳过 API 调用，直接复用上次成功的输出，节省 API 成本和时间。

| 功能项 | 描述 |
|---|---|
| 缓存匹配 | 基于 model + prompt + params 生成 hash，命中缓存时跳过 API 调用 |
| 缓存来源 | 缓存来自 `GenerationTask.output_payload` 中已成功的历史记录 |
| 缓存命中日志 | 命中缓存时记录结构化日志（cache_hit = true + 原始 task_id） |

**验收标准**：

- AC-M9-07：相同 prompt + model + params 的第二次请求直接返回缓存结果，耗时 < 1 秒
- AC-M9-08：修改任意参数（如 size、model）后不再命中缓存，发起新的 API 调用
- AC-M9-09：缓存命中在任务日志中有明确标记 `[缓存命中]`

### 6.4 FR-M9-04：成本追踪（Cost Tracking）

**需求描述**：每次 AI 生成任务的 usage / duration / provider 元数据存入 `output_payload`，支持后续成本分析。

| 功能项 | 描述 |
|---|---|
| usage 元数据 | 记录 token usage（prompt_tokens / completion_tokens / total_tokens） |
| duration 元数据 | 记录 API 调用耗时（`duration_ms`） |
| provider 元数据 | 记录实际调用的 provider_id 和 model_id |
| 存储位置 | 所有元数据存入 `GenerationTask.output_payload` JSON 字段 |

**验收标准**：

- AC-M9-10：每个成功任务的 `output_payload` 包含 `usage`、`duration_ms`、`provider_id` 字段
- AC-M9-11：成本数据可通过任务列表 API 查询到

### 6.5 FR-M9-05：任务队列监控 API

**需求描述**：提供队列健康状态查询接口，方便运维和调试。

| 功能项 | 描述 |
|---|---|
| 队列状态接口 | `GET /api/tasks/queue/status` 返回队列当前状态 |
| 返回内容 | 包含 pending 任务数、running 任务数、max_concurrent、running_slip 等运行时指标 |

**验收标准**：

- AC-M9-12：`GET /api/tasks/queue/status` 返回 200 且包含 `pending_count`、`running_count` 字段
- AC-M9-13：无活跃任务时各计数为 0，不报错

### 6.6 FR-M9-06：数据保留策略（Data Retention）

**需求描述**：自动清理过期的日志和终态任务数据，防止数据库和磁盘无限增长。

| 功能项 | 描述 |
|---|---|
| 日志自动清理 | 超过保留期限（默认 30 天）的结构化日志自动删除 |
| 终态任务清理 | 超过保留期限的 `succeeded` / `failed` / `cancelled` 任务自动归档或删除 |
| 清理时机 | 应用启动时执行一次，之后每 24 小时执行一次 |
| 清理日志 | 每次清理记录删除条数，写入结构化日志 |

**验收标准**：

- AC-M9-14：启动后日志中有一条清理记录，显示删除了多少条过期日志和任务
- AC-M9-15：保留期限可通过配置调整（`config.yaml` 中的 `retention_days`）
- AC-M9-16：清理操作不影响进行中的任务
