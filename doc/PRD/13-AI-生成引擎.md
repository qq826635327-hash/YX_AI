# PRD-13 · AI 生成引擎

> **版本**：v1.0 · **Owner**：架构负责人 · **关联**：`doc/开发手册/05-Provider与ComfyUI对接.md`

## 1. 模块定位

AI 生成引擎是本产品的**核心差异化模块**。它通过 Provider 抽象层对接多家 AI 服务商，将统一的 `StandardGenerateRequest` 翻译为各 Provider 的特定 API 格式，并将结果回流到 Asset 表。

## 2. 业务规则

### BR-1：Provider 抽象（translate/parse 模式）
```python
class ProviderHandler:
    def translate(req: StandardGenerateRequest) -> (url, payload, headers): ...
    def parse(response_body) -> StandardGenerateResult: ...
```
- 子类仅需实现 `translate` 与 `parse` 两个方法
- 父类提供 `generate_image()` / `generate_video()` 默认实现

### BR-2：注册机制
- 使用 `@register(provider_kind)` 装饰器注册 handler
- `provider_kind` ∈ {agnes, openai, fal, replicate, comfyui, custom}
- 启动时扫描 `providers/` 目录自动注册

### BR-3：任务生命周期
```
[pending] ──create_task 启动──▶ [running, progress 5%]
                                                      │
                          ┌──── 成功 ────────────────▶ [succeeded, progress 100%]
                          │                              │
                          │                              ▼
                          │                         Asset 创建
                          │
                          └──── 失败 ────────────────▶ [failed]
                                                         │
                                                         ▼
                                                    支持 retry（最多 3 次）
```

### BR-4：并发控制
- 全局信号量限制 4 个任务并发执行
- 超出部分排队（status = queued）

### BR-5：参考图注入（Reference Image Injection）
- 根据 `config.yaml` 的 `reference_prompt` 配置，自动将引用实体主图注入 Prompt
- 4 种预设：full / first_frame_only / first_and_last_frame / none

### BR-6：视频生成特殊规则（Agnes Video V2.0）
- 异步任务 API：POST 创建 → 轮询 GET 查状态
- 帧数必须符合 `8n + 1` 公式
- 支持 text-to-video / image-to-video / 关键帧动画

### BR-7：图床中转
- 部分 Provider 要求公开 URL（非本地路径）
- 通过 `image_hosting_service` 上传到图床（当前默认 boltp / 闪电图床，兼容 smms / superbed），SHA256 去重缓存

### BR-8：智能降级（Provider Fallback）
- 当主 Provider API 调用失败时，自动降级到备用 Provider（`fallback_provider_id`）
- 实现为 while-True 重试循环，最多 1 次 fallback（主 → 备 → 失败）
- 降级时记录结构化日志（`phase=system, event_type=provider_fallback`）
- 备用 Provider 必须在配置中标记为 enabled，否则跳过

### BR-9：Prompt 缓存
- 使用 SHA-256 哈希计算 `model + prompt + params + sorted(reference_asset_ids)` 的指纹
- 若同一项目内存在相同哈希的已成功任务，直接复制其输出文件，跳过 API 调用
- 缓存命中时任务状态立即标记为 `succeeded`，日志记录 `event_type=cache_hit`
- 缓存键存储在 `task.cache_key` 字段
- 重试任务自动设置 `force=True` 跳过缓存，确保重新调用 API
- `force` 字段：用户可通过「强制重新生成」复选框强制跳过缓存

### BR-10：成本追踪（StandardGenerateResult metadata）
- 每次 API 调用的元数据（usage、duration_ms、provider_kind、model）存储在 `StandardGenerateResult.metadata` 中
- 通过 `handler._last_result` 在任务完成后写入 `task.output_payload`
- 元数据包含：`usage`（token 用量 / API credits）、`duration_ms`（调用耗时）、`provider_kind`、`model`

### BR-11：Provider _last_result 模式
- `base.py` 中的 ProviderHandler 在 `generate_image()` / `generate_video()` 执行后，将完整 `StandardGenerateResult` 存储在实例属性 `_last_result` 上
- 任务执行器通过 `handler._last_result` 在 post-call 阶段获取元数据（如 usage、duration_ms）
- 避免修改 `generate_image()` 的返回签名，保持 translate/parse 模式的简洁性

### BR-12：智能自动重试（下载失败时复用 API URL）
- API 成功返回 URL 后，将 URL 列表保存到 `input_payload._result_urls`
- 后续步骤（下载/保存）失败时，系统自动重试下载（不重新调 API），条件：
  - `_result_urls` 存在（API 已成功）
  - `auto_retry_count < auto_retry_max_attempts`（默认 3 次）
  - 任务年龄 < `task_max_age_minutes`（默认 30 分钟）
- 自动重试延迟：`10 * retry_count` 秒（10s, 20s, 30s）
- URL 过期检测：HTTP 403/410/404 时清除 `_result_urls`，下次重试重新调 API
- 手动重试时取消正在等待的自动重试协程（`cancel_auto_retry`），避免竞态
- 手动重试重置 `auto_retry_count=0`
- 服务重启时 `recover_orphan_tasks`：有 `_result_urls` 的孤儿任务自动重试下载
- 下载成功后清除 `_result_urls`

### BR-13：默认模型自动选取
- 当任务 `provider_id` 为空时，根据 `default_models` 配置自动查找匹配的 Provider 和模型
- 映射关系：`image_generation` → `default_image_model`，`video_generation` → `default_video_model`，`text_reasoning` → `default_text_model`
- 配置修改持久化到 `config.yaml`，重启后不丢失

#### UI 交互细节

- **目标预览**：显示当前生成目标的元信息（图片、名称、类型）
- **参数面板**（根据 Provider 能力动态渲染）：

  | 字段 | 类型 | 说明 |
  |------|------|------|
  | Prompt | textarea | 主提示词 |
  | 负向提示词 | textarea | 可选 |
  | 尺寸 | select | 1024x1024 等 |
  | 参考图 | multi-select | 选自项目内已生成素材 |
  | 模型 | select | 选自可用模型 |
  | 高级参数 | accordion | 温度、步数、guidance 等 |

- **底部按钮**：取消 + 生成（提交后端任务）
- **提交流程**：
  ```
  点击"生成" → 收集参数 → POST /api/generate
  → 后端创建 GenerationTask 记录 → asyncio.create_task 启动执行器
  → 执行器异步执行 → WebSocket 推送进度
  → 完成后创建 Asset 记录 → 更新目标实体的 image_asset_id
  ```
- **进度反馈**：提交后立即跳转到任务中心，通过 WebSocket 推送进度，实时显示当前任务状态
- **请求体**：
  ```json
  {
    "project_id": "uuid",
    "target_type": "shot_first_frame",
    "target_id": "uuid",
    "prompt": "string",
    "negative_prompt": "string",
    "size": "1024x1024",
    "reference_assets": ["uuid"],
    "provider_id": "uuid",
    "workflow_id": "uuid",
    "extra_params": {}
  }
  ```

## 6. 异常场景

| 场景 | 处理 |
|---|---|
| Provider 未注册 | 后端返回 501，前端提示"该 Provider 暂未实现" |
| API Key 无效 | 测试连接失败，GenerateDialog 禁用该 Provider |
| 下载超时（600s） | 任务 failed，保留已生成的远程资产，支持重试 |
| 信号量满（4 个并发已满） | 新任务进入 queued，任务中心显示"排队中" |
| 帧数不符合 8n+1 | 前端校验 + 后端校验，提示"视频帧数必须为 8n+1" |
| 本地 ComfyUI 服务器离线 | 任务 failed，日志记录 `phase=system, event_type=comfyui_offline` |
| 下载失败但 API 已成功 | 自动重试下载（复用 `_result_urls`），最多 3 次，30 分钟超时 |
| URL 过期（403/410/404） | 清除 `_result_urls`，下次重试重新调 API |
| 自动重试与手动重试竞态 | `cancel_auto_retry` 取消自动重试协程，手动重试优先 |
| 默认模型配置重启丢失 | `save_settings_to_yaml()` 持久化到 config.yaml |

## 7. StandardGenerateRequest 规格

```json
{
  "provider_kind": "agnes",
  "model": "agnes-image-2.1-flash",
  "prompt": "A young man in a black coat...",
  "negative_prompt": "low quality, blur",
  "size": "1024x1024",
  "reference_images": [
    {"url": "https://sm.ms/image/xxx.png", "role": "character"}
  ],
  "count": 1,
  "extra": {
    "seed": 12345,
    "style_hint": "realistic"
  }
}
```

## 8. StandardGenerateResult 规格

```json
{
  "status": "succeeded",
  "media": [
    {
      "type": "image",
      "url_or_path": "/tmp/xxx.png",
      "width": 1024,
      "height": 1024,
      "mime": "image/png"
    }
  ],
  "metadata": {
    "seed": 12345,
    "provider_request_id": "xxx",
    "duration_ms": 8500,
    "provider_kind": "agnes",
    "model": "agnes-image-2.1-flash",
    "usage": {
      "total_tokens": 0,
      "api_credits": 0.02
    }
  }
}
```

> `metadata` 中的 `usage`、`duration_ms`、`provider_kind`、`model` 字段由 ProviderHandler 的 `_last_result` 机制回填，任务执行器在 post-call 阶段将其写入 `task.output_payload`。
