# PRD-14 · 任务中心与日志

> **版本**：v1.0 · **Owner**：业务模块负责人 · **关联**：`doc/开发手册/07-日志监控开发指南.md`

## 1. 模块定位

任务中心是用户的**全局监控台**，日志监控是开发者的**诊断工具**。二者共同构成本产品的可观测性基础。

## 2. 业务规则

### BR-1：任务状态机
```
[pending] ─▶ [queued] ─▶ [running] ─┬─▶ [succeeded]
                                    │
                                    ├─▶ [failed] ──retry──▶ [pending]
                                    │
                                    └─▶ [cancelled]
```

### BR-2：任务归属
- 每个 GenerationTask 关联一个 `target_type` + `target_id`
- `target_type` ∈ {character, scene, prop, shot_first_frame, shot_last_frame, shot_video}

### BR-3：WebSocket 推送频道
| 频道 | 事件 | 消息体 |
|---|---|---|
| `/ws/tasks` | 任务创建 / 进度 / 完成 / 失败 / 取消 | `{event, task_id, status, progress, error}` |
| `/ws/script` | 解析进度 | `{stage, progress, stream_text}` |
| `/ws/logs` | ERROR/WARNING 日志 | `{level, message, timestamp, phase, trace_id}` |

### BR-4：日志分级与存储
- **应用日志**：Python logging → 文件（`logs/app.log`）+ WebSocket（ERROR/WARNING 仅推）
- **任务日志**：结构化记录到 TaskLog 表（含 phase / event_type / data_json / trace_id）
- **前端日志**：`logStore` 捕获 `console.error`，上限 500 条，环形缓冲

### BR-5：重连策略
- WebSocket 断线后指数退避（初始 1s，最大 30s）+ 随机抖动（jitter 0-1s）
- 重连成功后全量刷新对应 TanStack Query

#### UI 交互细节

- **状态过滤 Tab**：

  | Tab | 包含状态 |
  |-----|----------|
  | 全部 | 全部（默认） |
  | 进行中 | pending / running / queued |
  | 已完成 | succeeded |
  | 失败 | failed |
  | 已取消 | cancelled |

- **任务卡片**：
  - **状态图标**：带颜色背景圆圈 — 排队中（黄）/ 生成中（蓝，带 spinner 动画）/ 已完成（绿）/ 失败（红）/ 已取消（灰）
  - **任务信息**：目标类型 + 目标 ID 简写、状态标签、进度百分比（生成中时）、错误信息（失败时）
  - **进度条**：运行中 / 排队中显示
  - **操作按钮**：

    | 状态 | 可用操作 |
    |------|---------|
    | 运行中 | 取消 |
    | 失败 | 重试 |
    | 成功 | 查看详情 |
    | 全部 | 删除 |

- **批量操作**：批量取消（多选运行中任务）、批量重试（多选失败任务）
- **当前项目 vs 全部项目**：

  | 入口 | 范围 | URL 参数 |
  |------|------|----------|
  | 当前项目 | 仅该项目 | `?project_id=xxx` |
  | 全部项目 | 跨项目，可按项目过滤 | 无 |

- **实时同步**：WebSocket 推送新任务创建/进度更新/状态变化，5 秒轮询作为兜底
- **后端 API**：`GET /api/tasks?project_id=&status=&target_type=&page=&page_size=`、`POST /api/generate/{tid}/cancel`、`POST /api/generate/{tid}/retry`、`POST /api/generate/batch`

#### UI 交互细节

- **架构链路**：Python logging → WsLogHandler（仅 ERROR/WARNING）→ WebSocket `/ws/logs` → useLogWsSubscription → Zustand logStore（最多 500 条）→ LogViewer 浮窗
- **消息格式**：
  ```json
  {
    "type": "log",
    "data": {
      "level": "ERROR",
      "logger": "app.services.asset_service",
      "message": "...",
      "module": "asset_service",
      "lineno": 191,
      "func": "delete_asset"
    },
    "timestamp": "2026-06-20T14:33:21.123+08:00"
  }
  ```
- **浮窗操作**：
  - 关键字搜索：实时过滤 message / logger
  - 级别按钮：一键只显示 ERROR / WARNING / INFO / DEBUG
  - 自动滚动：默认开，用户向上滚动自动暂停；新日志到达时自动滚到最下方
  - 复制为 JSON：把当前列表复制成 JSON（方便贴给 AI）
  - 复制为文本：每行 `[时间] [级别] [来源/logger] 消息 + 上下文`
  - 清空：调用 `DELETE /api/logs/clear` 清空后端日志文件，同时清空前端缓存（仅开发模式可用）
  - 时间显示：当天显示 `HH:MM:SS`，跨天显示 `MM-DD HH:MM:SS`
  - 排序：按时间正序，最新日志在最下方
  - 阶段标签：日志行显示 `[系统]` `[校验]` `[参考图]` `[生成]` `[下载]`，颜色区分
  - 结构化摘要：API 调用日志显示 `provider | model | status | duration`，下载日志显示 `size | duration`
  - 展开详情：点击"详情"查看完整 `data_json`
- **历史日志 API**：`GET /api/logs`（参数：level / limit / offset / keyword）、`GET /api/logs/info`（文件元信息）、`DELETE /api/logs/clear`（仅开发模式）
- **任务结构化日志 API**：`GET /api/logs/tasks/{task_id}`（参数：level / phase / limit），返回按 phase 分组的结构化日志

## 4. 用户故事

- **US-14.1**：作为小北，我希望在任务中心一眼看到所有正在进行的生成任务，失败的任务能一键重试。
- **US-14.2**：作为开发者，我希望在应用内实时看到 ERROR 日志，不需要切换去终端。
- **US-14.3**：作为 AI 助手，我希望日志含 trace_id 与结构化字段，方便自动定位问题。

## 5. 验收标准

| AC | Given | When | Then |
|---|---|---|---|
| AC-14.1 | 同时有 4 个任务执行 | WebSocket 推送 | 任务卡片实时刷新进度（无手动 polling） |
| AC-14.2 | 任务失败 | 查看 LogViewer | 看到 ERROR 日志 + 红点 + trace_id |
| AC-14.3 | 网络断开 10 秒后恢复 | WebSocket 重连 | 前端自动重连并刷新所有任务状态 |
| AC-14.4 | 前端 store 日志超过 500 条 | 新日志进入 | 旧日志被环形缓冲淘汰，总条数 ≤ 500 |
| AC-14.5 | 点击"一键复制 JSON" | 粘贴到 AI 助手 | 可被直接解析为 JSON 对象 |

## 6. 异常场景

| 场景 | 处理 |
|---|---|
| WebSocket 长时间断线（> 5 分钟） | 触发 fallback polling（每 15s 调 `GET /api/tasks`） |
| 任务日志写入失败 | 任务状态仍推进，但日志缺失，前端提示"日志不可用" |
| 前端 `console.error` 拦截异常 | 所有写入 `logStore` 的操作都 try/except 包裹，防止死循环 |
| 日志文件过大（> 100MB） | 启动时自动归档为 `.log.1`，新日志写入 `.log` |
