# PRD-16 · 资产管理与存储

> **版本**：v1.0 · **Owner**：业务模块负责人 · **关联**：`doc/开发手册/12-文件存储约定.md`

## 1. 模块定位

资产（Asset）是项目的**物理产物**，包含图片、视频、Prompt 文件。本模块负责：
- 资产的 CRUD 与文件存储
- DB ↔ 磁盘双向同步
- 图床中转（当前默认 boltp / 闪电图床，兼容 smms / superbed）

## 2. 业务规则

### BR-1：资产类别
| category | 用途 |
|---|---|
| character | 角色主图与附加图 |
| scene | 场景主图与附加图 |
| prop | 道具主图与附加图 |
| first_frame | 分镜首帧 |
| last_frame | 分镜末帧 |
| shot_video | 分镜视频 |
| upload | 用户上传的参考图 |

### BR-2：文件命名约定
- 图片：`{category}_{asset_id}.{ext}`（如 `character_a1b2c3d4.png`）
- 视频：`{category}_{asset_id}.{ext}`（如 `shot_video_a1b2c3d4.mp4`）
- Prompt：`{shot_no}_{section}.txt`（如 `01_first_frame.txt`）

### BR-3：目录结构
```
data/projects/{project_name}_{uuid}/
├── script/
│   └── script_v{N}.txt
├── 角色/{character_name}_{uuid}/{images|videos|prompts}/
├── 场景/{scene_name}_{uuid}/{images|videos|prompts}/
├── 道具/{prop_name}_{uuid}/{images|videos|prompts}/
├── 分镜/第{N}集_{shot_no}/{images|videos|prompts}/
└── assets/  # 临时下载区
```

### BR-4：双向同步规则
- **DB → 磁盘**：删除 DB 记录但文件存在的孤儿文件
- **磁盘 → DB**：扫描目录，发现未注册文件，创建 Asset 记录
- **SHA256 校验**：同一文件不重复创建

### BR-5：上传限制
- 单文件 ≤ 200MB
- 图片类型：png / jpg / jpeg / webp / gif
- 视频类型：mp4 / webm / mov / mkv
- 文件名清洗：去除特殊字符，转小写扩展名

### BR-6：主图关联
- `Character.image_asset_id` 等指向 Asset.id
- 设置主图：更新实体表的 `image_asset_id`
- 删除主图资产时，自动清空 `image_asset_id`

### BR-7：图床中转

#### 7.1 用途
- 部分 AI 模型（如视频生成模型）要求参考图使用**公网 URL**（非本地路径或 base64）
- 图床服务将本地图片上传到公网，获取可访问的 URL
- 是否需要图床由**模型能力声明**中的 `reference_images_need_url` 字段决定（在模型配置页可勾选）

#### 7.2 图床配置管理（数据库化）
- 图床配置存储在 `image_hosting_providers` 数据库表中，前端可自由 CRUD
- 支持 5 种图床类型：

| 类型 | 标识 | 说明 | API 地址 |
|------|------|------|----------|
| 闪电图床 | `boltp` | 国内访问快，默认 10MB 限制 | 自动填充 |
| SM.MS | `smms` | 老牌图床，5MB 限制 | 自动填充 |
| 聚合图床 | `superbed` | 多源聚合，10MB 限制 | 自动填充 |
| GitHub | `github` | 免费存储，通过 Contents API 上传 | 自动填充 |
| 自定义 | `custom` | 适配任何 multipart POST 图床 API | 用户填写 |

- 预设类型（boltp/smms/superbed/github）的 API 地址自动填充，用户只需填密钥
- 自定义类型需用户填写 API 地址、密钥、响应 URL 路径等
- Token 加密存储（Fernet），前端展示时脱敏
- 支持设置**默认图床**（`is_default = true`），生成任务时自动使用默认图床
- 支持启用/停用（`enabled`），停用的图床不会被选中
- 支持测试上传（上传一张 1x1 透明 PNG 验证配置是否正确）

#### 7.3 图床选择优先级
1. 数据库中 `is_default = true` 且 `enabled = true` 的图床
2. 数据库中第一个 `enabled = true` 的图床
3. `config.yaml` 中的 `image_hosting` 配置（向后兼容 fallback）

#### 7.4 上传缓存机制
- 上传结果缓存到 Asset 表的 `public_url` / `public_url_file_hash` 字段
- 每次上传前计算图片的 **SHA256 哈希值**
- 如果哈希值与 `public_url_file_hash` 一致且已有 `public_url` → **直接返回缓存 URL，跳过上传**
- 只有文件内容变更（哈希不同）或从未上传过，才真正执行上传
- **效果**：同一张参考图（如首帧/尾帧）多次用于不同任务时，只上传一次

#### 7.5 文件大小与自动压缩
- 每个图床类型有默认文件大小限制（如 SM.MS 5MB，GitHub 100MB）
- 上传前检查文件大小，超限则自动压缩为 JPEG（逐步降低质量直到满足限制）
- 压缩仍超限则缩小尺寸到 1920px 宽后重试
- 压缩使用临时文件，上传完成后自动清理

#### 7.6 GitHub 图床特殊逻辑
- 通过 GitHub Contents API 上传（Base64 编码）
- `extra_config` 需包含：`owner`（用户名）、`repo`（仓库名）、`branch`（分支，默认 main）、`path_prefix`（路径前缀，默认 assets）
- 文件存储路径：`{path_prefix}/{YYYYMMDD}/{filename}`
- 同名文件已存在时，自动获取已有文件的 download_url（不重复上传）
- 访问 URL 格式：`https://raw.githubusercontent.com/{owner}/{repo}/{branch}/{path}`

#### 7.7 自定义图床逻辑
- 通用 multipart POST 上传，适配任何图床 API
- `extra_config` 可配置：`file_field`（文件字段名，默认 file）、`extra_fields`（额外表单参数）、`url_path`（响应中 URL 的 JSON 路径，如 `data.url`）、`auth_format`（Authorization 格式，默认 `Bearer {token}`）
- 响应中自动尝试常见 URL 路径：`data.url`、`data.public_url`、`url`、`data.link`、`data.image`

#### 7.8 启动迁移
- 首次启动时，自动将 `config.yaml` 中的图床配置迁移到数据库（幂等，仅执行一次）
- 迁移条件：数据库中无任何图床配置记录
- 迁移后，`config.yaml` 中的配置保留但不再作为主配置源

- **视觉结构**：网格布局的缩略图列表，顶部有标题、过滤/排序、批量操作；每个素材项含缩略图、文件名、上传时间、类型徽章

- **素材类型**：

  | 类型 | 支持格式 |

## 6. 异常场景

| 场景 | 处理 |
|---|---|
| 磁盘目录被用户手动删除 | 访问 Asset 文件时 404，画廊显示占位图 + "文件丢失"徽章 |
| DB 记录删除但文件未删 | 同步按钮清理磁盘文件 |
| 文件被占用无法删除 | `try/except PermissionError`，DB 记录删除，残留文件留给同步清理 |
| 图床上传失败 | 任务 failed，日志记录 `phase=upload, error=hosting_failed`，重试按钮可用 |
| 同名文件重复上传 | SHA256 校验，命中则复用，否则追加序号 |
