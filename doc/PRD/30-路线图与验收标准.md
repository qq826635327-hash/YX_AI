## 5. 验收标准（整体）

### 5.1 测试覆盖率目标

| 类别 | 目标 | 当前基线 |
|---|---|---|
| 后端单元测试 | 90% | 75%（估算） |
| 后端 API 测试 | 100%（所有端点） | 85% |
| 前端页面测试 | 80%（核心页面） | 少量（vitest 已集成，持续补充） |
| E2E 测试 | 10 条核心路径 | 5 条 |
| WebSocket 测试 | 100%（3 个频道） | 80% |

### 5.2 性能基线（Phase 3 末）

| 指标 | 目标 |
|---|---|
| 启动时间 | < 10s（含 WAL checkpoint + 自动备份） |
| 列表 API P95 | < 300ms |
| 剧本解析耗时（5000 字） | < 180s |
| 单任务生成耗时（1024x1024 图） | < 30s |
| 单任务生成耗时（5s 视频） | < 180s |
| 优雅关闭耗时 | < 120s（drain_tasks 超时上限） |
| 数据库备份耗时 | < 5s（.backup API 快照） |

### 5.3 v1.1 新增功能验收标准

| 功能 | AC（Given-When-Then） |
|---|---|
| **数据库备份** | Given 应用启动 → When lifespan startup 执行 → Then `data/backups/` 下生成带时间戳的备份文件 |
| **手动备份** | Given 用户调用 `POST /api/config/backup` → When 请求成功 → Then 返回备份文件路径，过期备份自动清理 |
| **数据保留** | Given 存在 > 30 天的 task_logs → When 应用启动 → Then 过期记录被删除，日志记录清理数量 |
| **WAL 检查点** | Given WAL 文件 > 0 → When 应用启动 → Then WAL 文件收缩，主数据库文件包含最新数据 |
| **优雅关闭** | Given 有 in-flight 任务 → When 服务器关闭 → Then 任务在 120s 内完成，超时则保留 running 状态 |
| **信号量重置** | Given --reload 重启 → When 新进程启动 → Then Semaphore 计数重置为初始值，残留 running 任务标记 failed |
| **Prompt 缓存** | Given 两次相同 Prompt 请求 → When cache_key 匹配 → Then 第二次请求命中缓存，不重复调用 Provider |
| **成本追踪** | Given 任务完成 → When 查询 GenerationTask → Then output_payload 包含 cost/tokens_used 字段 |
| **队列可观测性** | Given 调用 `GET /api/tasks/queue/status` → When 有 in-flight 任务 → Then 返回准确的信号量计数和队列状态 |
| **智能降级** | Given 主 Provider 调用失败 → When `smart_fallback=true` 且存在启用备用 Provider → Then 自动切换并完成生成 |

## 6. 关键风险跟踪

| 风险 | 触发条件 | 缓解措施 | 当前状态 |
|---|---|---|---|
| SQLite WAL 性能瓶颈 | 项目资产 > 10000 | Phase 4 评估迁移 PG；启动 WAL TRUNCATE 检查点 | 🟢 低 |
| Agnes API 稳定性 | 连续 3 天成功率 < 70% | `fallback_provider` 智能降级 + Provider 健康检查 | 🟡 中 |
| ComfyUI 集成延期 | M3.4 超期 1 周 | 缩小范围，仅支持 image 工作流 | 🟡 中 |
| 文档脱节 | 每周一次审计发现不一致 > 5 处 | 强制 PRD 修改同步流程 | 🟢 低 |
| 服务器意外中断 | 断电 / crash / OOM | `drain_tasks()` 优雅关闭 + `reset_task_state()` 启动恢复 | 🟢 低（已缓解） |
| 数据库损坏 | WAL 文件异常 / 磁盘满 | 启动自动备份 + 可配置保留策略 + 手动备份 API | 🟢 低（已缓解） |
