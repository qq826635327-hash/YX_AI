# 日志监控（AI 开发助手专用）

> 该模块的目标：让 AI 开发助手（或开发者）不需要 ssh 到服务器、不需要翻 `logs/backend.log`，就能在浏览器里直接看到前后端错误。

## 模块概览

| 项 | 描述 |
| --- | --- |
| 功能入口 | 右下角悬浮按钮（带未读红点） |
| 数据来源 | ① 实时：`/ws/logs` WebSocket ② 历史：`GET /api/logs` |
| 关闭方式 | 关闭按钮 / `Ctrl+Shift+L` 快捷键 |
| 默认级别 | 实时推送：WARNING/ERROR；历史可按级别过滤 |
| 缓存上限 | 前端最多保留 500 条；按时间正序排列，最新日志在列表最下方 |

## 二、消息格式

### 后端推送给前端

```json
{
  "type": "log",
  "data": {
    "level": "ERROR",
    "logger": "app.services.asset_service",
    "message": "2026-06-20 14:33:21 [app.services.asset_service] ERROR: 删除文件失败: ...",
    "module": "asset_service",
    "lineno": 191,
    "func": "delete_asset"
  },
  "timestamp": "2026-06-20T14:33:21.123+08:00"
}
```

> 时间戳使用带本地时区偏移的 ISO8601（如 `+08:00`），避免 UTC/本地时间混淆。后端 `logs.py` 会把日志文件里无时区的 `%(asctime)s` 解释成本地时间。

### 客户端命令

- `ping`：心跳
- `clear`：通知前端清空自己的本地缓存（可选）

## 三、API

### `GET /api/logs`

读取 `logs/backend.log` 历史日志（倒序）。

Query 参数：

| 参数 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| `level` | enum | null | DEBUG/INFO/WARNING/ERROR/CRITICAL |
| `limit` | int | 200 | 最多返回条数（1-2000） |
| `offset` | int | 0 | 倒序偏移 |
| `keyword` | string | null | message 子串过滤（不区分大小写） |

返回：

```json
{
  "data": {
    "total": 1234,
    "returned": 200,
    "file": "D:\\影序AI\\backend\\logs\\backend.log",
    "entries": [
      {
        "timestamp": "2026-06-20T14:33:21+08:00",
        "level": "ERROR",
        "logger": "app.services.asset_service",
        "message": "删除文件失败: ..."
      }
    ]
  }
}
```

> 注意：所有 `/api/logs/*` 端点都使用统一响应包装 `{ "data": ..., "message": ... }`，与项目其他 API 一致。

### `GET /api/logs/info`

返回日志文件元信息（路径、大小、最后修改时间）。

### `DELETE /api/logs/clear`

清空日志文件（仅开发模式，生产环境返回 403）。

### `GET /api/logs/tasks/{task_id}`（2026-06-21 新增）

获取任务的结构化日志（从 DB 读取），包含 API 请求/响应/耗时等完整信息。

Query 参数：

| 参数 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| `level` | enum | null | DEBUG/INFO/WARN/ERROR |
| `phase` | string | null | 按阶段过滤（generate/download/ref_collect/validate/system） |
| `limit` | int | 200 | 最多返回条数（1-1000） |

返回：

```json
{
  "data": {
    "task_id": "abc123",
    "trace_id": "a3f8c2d1e4b0",
    "total": 9,
    "entries": [
      {
        "id": "...",
        "task_id": "abc123",
        "level": "INFO",
        "message": "[generate] POST https://apihub... → 200 | 72340ms",
        "phase": "generate",
        "event_type": "api_request",
        "data_json": {
          "provider_kind": "agnes",
          "model": "agnes-image-2.1-flash",
          "response_status": 200,
          "duration_ms": 72340
        },
        "trace_id": "a3f8c2d1e4b0",
        "created_at": "2026-06-21T15:36:30+08:00"
      }
    ]
  }
}
```

## 四、前端使用

### 自动接入

无需业务代码改动。`LogViewer` 已在 `MainLayout` 挂载，`useLogWsSubscription` 已在全局订阅 `logsWs`。

### 手动记录日志

```ts
import { logger } from "@/stores/logStore";

// 记录后端 / API / 前端错误
logger.error("图片生成失败", "api", "GenerateDialog", { prompt: "...", status: 500 });
logger.warn("WS 断开，准备重连", "ws", "useTaskWsSubscription");
logger.info("用户点击了生成按钮", "frontend", "GenerateDialog");
logger.debug("参数调试", "frontend", "X", { a: 1 });
```

### LogViewer 浮窗操作

- 关键字搜索：实时过滤 message / logger
- 级别按钮：一键只显示 ERROR / WARNING / INFO / DEBUG
- 自动滚动：默认开，用户向上滚动自动暂停；新日志到达时自动滚到最下方
- 复制为 JSON：把当前列表复制成 JSON（方便贴给 AI）
- 复制为文本：每行 `[时间] [级别] [来源/logger] 消息 + 上下文`
- 清空：调用 `DELETE /api/logs/clear` 清空后端日志文件，同时清空前端缓存（仅开发模式可用）
- 时间显示：当天显示 `HH:MM:SS`，跨天显示 `MM-DD HH:MM:SS`
- 排序：按时间正序，最新日志在最下方（类似 `tail -f`）
- 详情：每行右侧"详情"按钮可展开 context 字段
- **阶段标签**（2026-06-21 新增）：日志行显示阶段标签 `[系统]` `[校验]` `[参考图]` `[生成]` `[下载]`，颜色区分
- **结构化摘要**（2026-06-21 新增）：API 调用日志显示 `provider | model | status | duration`，下载日志显示 `size | duration`
- **展开详情**（2026-06-21 新增）：点击"详情"查看完整 `data_json`（API 请求/响应/耗时等）

## 五、注意事项

1. **后端日志等级**：默认 INFO 级别。Python logging 的传播是按 logger name 匹配的，根 logger 配 INFO 就够用了。WS Handler 只取 ERROR/WARNING，避免刷屏。
2. **事件循环注入**：`ws_log_handler.set_loop(...)` 必须在 `lifespan` 启动后调用，否则跨线程广播会失败。`main.py` 已处理。
3. **FileHandler 锁**：清空日志用的是"截断"而不是"删除"，避免 Python 的 `FileHandler` 句柄失效。
4. **错误日志不要死循环**：`WsLogHandler.emit` 内部任何异常都被吃掉，避免日志广播本身崩溃导致整个 app 挂掉。
5. **生产环境**：`DELETE /api/logs/clear` 在 production 环境被拒绝；WS 推送依然开启（不需要额外鉴权，因为日志本身就不含敏感信息；如果有顾虑，可以加 BasicAuth 鉴权或单独开关）。
