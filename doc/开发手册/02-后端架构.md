# 02 · 后端架构

> 完整讲解后端技术栈和设计决策。AI 改后端代码前必看。

## 1. 技术栈

| 组件 | 选型 | 原因 |
| --- | --- | --- |
| Web 框架 | FastAPI ≥0.110 | 自动 OpenAPI、依赖注入、原生 WebSocket |
| ORM | SQLModel 0.0.x | 同时拿 Pydantic + SQLAlchemy，写法简洁 |
| 数据库 | SQLite (WAL) | 零部署；WAL 模式支持多读单写 |
| 任务队列 | asyncio.Semaphore + create_task | 轻量级异步任务调度，Semaphore 限流（最大 4 并发） |
| 配置 | PyYAML + 环境变量 | 文件配置 + 生产环境变量覆盖 |
| 加密 | cryptography.Fernet | API Key 必须加密存储 |
| 静态服务 | FastAPI StaticFiles | 生产模式直接托管 `frontend/dist` |
| 日志 | Python logging + 自定义 WsLogHandler | 标准库 + WS 广播扩展 |

## 2. 入口与生命周期

### `main.py` 启动流程

```python
# 1. 配置日志（同时输出到 stdout 和 logs/backend.log）
_setup_logging()

# 2. lifespan 钩子（应用启动/关闭）
@asynccontextmanager
async def lifespan(app: FastAPI):
    clear_settings_cache()       # 防止旧配置残留
    reset_engine()               # 释放旧 DB 连接
    ws_log_handler.set_loop(asyncio.get_running_loop())  # 注入事件循环
    set_ws_manager(manager)      # 注入 WS 连接管理器
    init_db()                    # 创建表
    validate_security(settings)  # 生产密钥校验
    yield
    # 关闭钩子

# 3. 中间件（按添加顺序反作用于响应）
app.add_middleware(CacheControlMiddleware)  # API 响应加 no-cache
app.add_middleware(BasicAuthMiddleware)     # 可选 BasicAuth
app.add_middleware(CORSMiddleware)          # 放最外层（保证 401 也有 CORS 头）

# 4. 路由
app.include_router(api_router)    # /api/...
app.include_router(ws_router)     # /ws/...
```

### 关键设计点

1. **`reset_engine()` 在启动时调用**：避免 Windows 下进程未完全退出导致旧 SQLAlchemy 连接池残留。
2. **`ws_log_handler.set_loop()` 在 lifespan 中注入**：WS LogHandler 是从其他线程（HTTP 请求处理线程）emit 的，必须有主事件循环才能跨线程调度广播。
3. **`CacheControlMiddleware` 包在 BasicAuth 外面**：所有 API 响应永远不缓存。
4. **BasicAuth 默认关闭**：开发期不挡路；生产环境通过 `config.yaml` 开启。

## 3. 数据库（`db.py`）

### 引擎特性

- **WAL 模式**：`PRAGMA journal_mode=WAL` + `synchronous=NORMAL` + `foreign_keys=ON`
- **`check_same_thread=False`**：允许跨线程访问
- **`expire_on_commit=False`**：commit 后属性不失效

### 会话管理

```python
# FastAPI 依赖注入
def get_session() -> Iterator[Session]:
    with Session(get_engine(), expire_on_commit=False) as session:
        yield session

# 任务/脚本用
@contextmanager
def session_scope() -> Iterator[Session]:
    session = Session(get_engine(), expire_on_commit=False)
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()
```

**注意**：
- HTTP 请求用 `Depends(get_session)`（自动关闭）
- 后台任务必须用 `session_scope()`（手动管理）
- 任务里**不要在闭包中持有 session**，用一次开一次
- 任务执行器中尽量合并 session_scope，减少 session 开关次数（参考 `execute_task.py` 的多阶段模式）

### 模型基类（`models/base.py`）

```python
class IDMixin(SQLModel):
    id: str = Field(default_factory=gen_uuid, primary_key=True, max_length=36)

class TimestampMixin(SQLModel):
    created_at: datetime = Field(default_factory=utcnow, nullable=False, index=True)
    updated_at: datetime = Field(default_factory=utcnow, nullable=False)
```

**所有业务表都继承这两个 Mixin**。

## 5. 通用序列化（`core/serialization.py`）

**为什么需要**：SQLModel 模型转 JSON 时，`datetime` 默认会变成 ISO 字符串，但 Pydantic v2 + FastAPI 已经会自动转；这个工具主要是给**直接 `model_dump()` 的场景**（如业务层手动构造响应）。

```python
from app.core.serialization import serialize_model, serialize_models

data = serialize_model(character)  # → dict
data = serialize_models(characters)  # → list[dict]
```

自动处理的字段：
- `_DATE_FIELDS = ("created_at", "updated_at", "started_at", "finished_at", "last_used_at")`
- `datetime` → ISO 字符串（naive datetime 自动附加 UTC 时区）

**注意**：所有需要序列化模型的地方都应使用此工具，不要手动拼接 dict（如 `{"id": x.id, "created_at": x.created_at.isoformat()}`），避免日期格式不一致。

## 5.5 全链路追踪（`core/trace.py`）

基于 `contextvars` 的 trace_id 追踪，串联同一任务的多步操作（ref_collect → generate → download → save）。

```python
from app.core.trace import new_trace_id, get_trace_id, clear_trace_id

# 任务开始时分配
tid = new_trace_id()  # → "a1b2c3d4e5f6"

# 任意位置获取
current_tid = get_trace_id()

# 任务结束时清理
clear_trace_id()
```

**用途**：`execute_task.py` 在任务开始时调用 `new_trace_id()`，后续 `log_api_call` / `log_ref_collect` / `log_download` 自动携带 trace_id，方便在日志中追踪同一任务的完整生命周期。

## 5.6 Provider 标准化类型（`schemas/provider_types.py`）

所有 Provider 统一的输入/输出模型，解耦厂商差异：

```python
class StandardGenerateRequest(BaseModel):

    size: str | None = None
    reference_images: list[str] = []  # Data URI Base64 列表

class ModelCapabilities(BaseModel):

    image_to_image: bool = False
```

**translate/parse 模式**：Provider Handler 只需实现 `translate()`（标准请求→厂商API请求）和 `parse()`（厂商响应→标准结果），`generate_image()` 由基类统一处理。

## 6. WebSocket 架构（`ws/`）

### 连接管理器

```python
class ConnectionManager:
    # channel → set[WebSocket]
    async def connect(self, ws, channel="default"): ...
    def disconnect(self, ws, channel="default"): ...
    async def broadcast(self, channel, message: dict): ...
    async def send_personal(self, ws, message: dict): ...
```

### 频道

| 频道 | 路径 | 触发场景 |
| --- | --- | --- |
| tasks | `/ws/tasks` | 任务创建/进度/完成/失败 |
| script | `/ws/script` | 剧本解析进度 |
| logs | `/ws/logs` | 后端 ERROR/WARNING 日志 |

三个频道共用 `_ws_channel_handler()` 通用处理函数，减少重复代码。

### 客户端消息

- `ping` → 回 `pong`
- `clear`（仅 logs 频道）→ 回 `log.cleared`

### 跨线程广播

```python
# 业务代码（可能在 worker 线程）调用：
async def push_task_progress(task_id, progress, **extra):
    await manager.broadcast("tasks", make_message("task.progress", {...}))
```

`async` 函数只能在事件循环里用。**后台任务里调用 ws push 的方式**：
- `execute_task.py` 是 `async def`，通过 `asyncio.create_task()` 在主事件循环中启动
- 任务内部可以直接 `await push_task_progress(...)` 等 WS 推送函数
- 并发控制：`asyncio.Semaphore(_MAX_CONCURRENT_TASKS=4)` 限流
- 任务引用保存在 `set[asyncio.Task]` 中，防止 GC 回收，完成后自动移除

## 7. 日志（详见 `07-日志监控开发指南.md`）

- `main.py._setup_logging()` 初始化：根 logger + console + FileHandler + WsLogHandler

## 8. 错误处理约定

### API 异常

FastAPI 路由里**用 `HTTPException`**，**不要直接 `raise Exception`**：

```python
# ✓ 正确
raise HTTPException(status_code=404, detail=f"项目 {project_id} 不存在")

# ✗ 错误
raise Exception("项目不存在")
```

`HTTPException` 会自动转为标准 JSON 响应 `{ "error": "<status_name>", "message": "<detail>", ... }`。

### 后台任务异常

`execute_task` 里的 try-except 会：
1. `logger.error(...)` 写入日志
2. 任务状态改为 `failed`
3. `push_task_failed(...)` 推送 WS

**不要让异常穿透到调用方** —— 调用方（`api/generate.py`）只负责创建任务和启动异步执行，不捕获任务内部异常。
