## 4. 数据请求（TanStack Query）

### 通用模式

```typescript
// hooks/useApi.ts / useProjects.ts / useBusiness.ts
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";

export function useProjects() {
  return useQuery({
    queryKey: ["projects"],
    queryFn: () => projectsApi.list().then(d => d.data),
  });
}

export function useCreateProject() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: projectsApi.create,
    onSuccess: () => qc.invalidateQueries({ queryKey: ["projects"] }),
  });
}
```

### 失效策略

| 场景 | 失效的 key |
| --- | --- |
| 创建/更新/删除 Project | `["projects"]` |
| 创建/更新/删除 Character/Scene/Prop | `["business", "<entity>", projectId]` + `["projects"]`（统计数） |
| 创建/更新/删除 Shot | `["business", "shots", episodeId]` + `["projects"]` |
| 任务进度/完成/失败（WS 推送） | `["tasks"]` |
| 剧本解析（WS 推送） | `["script", projectId]` |
| 项目详情页 | 任何实体变更时 `["project", projectId]` |

### `staleTime` 和 `retry`

`main.tsx` 里：

```typescript
new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 0,           // 每次都重新拉
      retry: 1,
      refetchOnWindowFocus: false,
    },
  },
});
```

## 5. HTTP 客户端（`api/client.ts`）

### `ky` 封装

```typescript
export const http = ky.create({
  prefixUrl: "/api",
  timeout: 60000,
  retry: { methods: ["get", "head"], limit: 2 },
  hooks: {
    afterResponse: [/* 4xx/5xx → ApiError + 写日志 */],
  },
});
```

### ApiError

```typescript
export class ApiError extends Error {
  status: number;
  body?: ApiErrorResponse;
}
```

**4xx 视为 warning，5xx 视为 error**，自动写入 logStore（前端可观察）。

### 响应 unwrap

后端统一返回 `{ data: T, message?: string }`（分页时 `data: { items, total, page, page_size }`）。

- `unwrap<T>()`：返回 `data` 字段
- `unwrapPaginated<T>()`：返回分页对象
- `unwrapFull<T>()`：返回完整 `{ data, message }`

## 6. WebSocket 客户端（`api/websocket.ts`）

### WsClient 类

```typescript
class WsClient {
  constructor(path: string) { ... }   // "/ws/tasks"
  connect(): void;                     // 自动重连（指数退避，最多 30 秒）
  on(handler): () => void;             // 订阅；返回 unsubscribe
  close(): void;
}
```

### 单例

```typescript
export const tasksWs = new WsClient("/ws/tasks");
export const scriptWs = new WsClient("/ws/script");
export const logsWs = new WsClient("/ws/logs");
```

**所有 WS 单例都在 `main.tsx` 启动时 `.connect()`，整个 app 生命周期保持连接。**

### 心跳

每 30 秒发一次 `ping`；服务端回 `pong` 或 `heartbeat`。

### 重新连接

指数退避：1s, 2s, 4s, 8s, 16s, 30s（封顶）。

## 14. TanStack Query 约定

> 本节汇总所有 Query Key、Mutation Invalidation、轮询策略和命名规范。AI 新增/修改数据请求时必须遵守。

### 14.1 Query Key 注册表

所有 Query Key 均通过 `*Keys` 工厂对象生成（如 `projectKeys`、`charKeys`），**禁止手写裸数组**。

| Query Key | Hook | 描述 | 自动刷新 |
| --- | --- | --- | --- |
| `["script", projectId]` | `useScript` | 获取项目剧本文本 | 否（WS 推送触发） |
| `["tasks", project_id?, status?, target_type?, target_id?, page?, page_size?]` | `useTasks` | 任务列表（分页、筛选） | 是（轮询 + WS） |
| `["pending-tasks", projectId, targetType, targetId]` | `usePendingTasks` | 指定实体的进行中任务 | 是（轮询） |
| `["task-poll", taskId]` | `useTaskPolling` | 单个任务状态轮询 | 是（轮询） |
| `["provider-capabilities", providerId, isComfyui]` | `useProviderCapabilities` | Provider 支持的模型/参数 | 否 |
| `["providers", enabled?]` | `useProviders` | AI Provider 列表 | 否 |
| `["workflows", params?]` | `useWorkflows` | ComfyUI 工作流映射列表 | 否 |
| `["system-config"]` | `useSystemConfig` | 系统全局配置 | 否 |
| `["characters", "list", projectId, charType?]` | `useCharacters` | 角色列表（可按类型过滤） | 否 |
| `["characters", "detail", projectId, id]` | `useCharacter` | 单个角色详情 | 否 |
| `["scenes", "list", projectId]` | `useScenes` | 场景列表 | 否 |
| `["scenes", "detail", projectId, id]` | `useScene` | 单个场景详情 | 否 |
| `["props", "list", projectId]` | `useProps` | 道具列表 | 否 |
| `["props", "detail", projectId, id]` | `useProp` | 单个道具详情 | 否 |
| `["episodes", "list", projectId]` | `useEpisodes` | 剧集列表 | 否 |
| `["shots", episodeId]` | `useShots` | 某剧集下的分镜列表 | 否 |
| `["shot-references", shotId]` | `useShotReferences` | 分镜关联的角色/场景/道具 | 否 |
| `["assets", "target", projectId, targetType, targetId]` | `useAssets` | 按实体查素材列表 | 否（WS 推送触发） |
| `["projects", "list", params?]` | `useProjects` | 项目列表（分页、搜索） | 否 |
| `["projects", "detail", id]` | `useProject` | 单个项目详情 | 否 |

**Key 工厂对象一览**（定义在各自的 hooks 文件中）：

```typescript
// useProjects.ts
export const projectKeys = {
  all: ["projects"],
  list: (params?) => [...projectKeys.all, "list", params],
  detail: (id) => [...projectKeys.all, "detail", id],
};

// useBusiness.ts
export const charKeys = {
  all: ["characters"],
  list: (projectId) => [...charKeys.all, "list", projectId],
  detail: (projectId, id) => [...charKeys.all, "detail", projectId, id],
};
export const sceneKeys = { /* 同 charKeys 模式 */ };
export const propKeys  = { /* 同 charKeys 模式 */ };
export const episodeKeys = {
  all: ["episodes"],
  list: (projectId) => [...episodeKeys.all, "list", projectId],
  shots: (episodeId) => ["shots", episodeId],
};
export const assetKeys = {
  all: ["assets"],
  byTarget: (projectId, targetType, targetId) =>
    [...assetKeys.all, "target", projectId, targetType, targetId],
};
```

### 14.2 Invalidation 规则

#### Mutation → Query 失效映射

| Mutation | Invalidates | 触发时机 |
| --- | --- | --- |
| `useSaveScript` / `useParseScript` | `["script", projectId]` | 保存剧本、启动解析 |
| `useGenerate` / `useRetryTask` / `useCancelTask` / `useClearTasks` / `useBatchGenerate` | `["tasks", ...]` | 提交/重试/取消/清理/批量生成任务 |
| `useCreateProvider` / `useUpdateProvider` / `useDeleteProvider` | `["providers", ...]` | Provider CRUD |
| `useCreateWorkflow` / `useUpdateWorkflow` / `useDeleteWorkflow` | `["workflows", ...]` | 工作流映射 CRUD |
| `useUpdateLLMConfig` | `["system-config"]` | 修改 LLM 配置 |
| `useCreateCharacter` / `useUpdateCharacter` / `useDeleteCharacter` | `["characters", "list", projectId]` | 角色 CRUD |
| `useCreateScene` / `useUpdateScene` / `useDeleteScene` | `["scenes", "list", projectId]` | 场景 CRUD |
| `useCreateProp` / `useUpdateProp` / `useDeleteProp` | `["props", "list", projectId]` | 道具 CRUD |
| `useCreateEpisode` / `useDeleteEpisode` | `["episodes", "list", projectId]` | 剧集增删 |
| `useUpdateShot` | `["shots", ...]` | 编辑分镜 |
| `useCreateShot` / `useDeleteShot` | `["shots", ...]` + `["episodes", "list", projectId]` | 分镜增删（同时刷新剧集列表，因为统计数可能变化） |
| `useAddShotCharacters` / `useRemoveShotCharacter` | `["shot-references", shotId]` | 分镜↔角色关联变更 |
| `useAddShotScenes` / `useRemoveShotScene` | `["shot-references", shotId]` | 分镜↔场景关联变更 |
| `useAddShotProps` / `useRemoveShotProp` | `["shot-references", shotId]` | 分镜↔道具关联变更 |
| `useDeleteAsset` | 无自动失效 | 素材删除后由 WS 推送触发刷新 |
| `useCreateProject` / `useDeleteProject` | `["projects", ...]`（all 前缀） | 项目增删 |
| `useUpdateProject` | `["projects", "detail", id]` + `["projects", ...]`（all 前缀） | 项目更新（同时刷新列表和详情） |

#### WS 推送 → Query 失效映射（`useWs.ts`）

| WS 事件 | Invalidates | 说明 |
| --- | --- | --- |
| `task.created` / `task.progress` | `["tasks"]` | 新任务或进度更新，刷新任务列表 |
| `task.completed` / `task.failed` | `["tasks"]` + `["assets"]` + 对应实体 key | 按 `target_type` 精确刷新：`shot_*` → `["shots"]`、`character` → `["characters"]`、`scene` → `["scenes"]`、`prop` → `["props"]`；未知类型全量刷新所有实体 |
| `script.completed` / `script.failed` / `script.parsing` | `["script", projectId]` | 剧本解析状态变更（仅处理当前项目消息） |

### 14.3 refetchInterval 策略

| Hook | 活跃时 | 空闲时 | 逻辑说明 |
| --- | --- | --- | --- |
| `useTasks` | **5 秒** | **30 秒** | 检查列表中是否有 `pending`/`running`/`queued` 状态的任务；有则 5s 快速轮询，否则 30s 慢轮询（WS 推送会即时 invalidate，所以慢轮询只是兜底） |
| `usePendingTasks` | **2 秒** | **停止**（`false`） | 用于详情页占位符显示；有进行中任务时 2s 轮询，全部完成/失败后停止轮询 |
| `useTaskPolling` | **2 秒** | **停止**（`false`） | 单任务轮询；`succeeded`/`failed`/`cancelled` 终态时自动停止 |

**设计思路**：WS 推送是主要的实时更新通道，轮询是兜底机制。活跃任务用短间隔保证用户体验，空闲时用长间隔（或停止）减少服务器压力。

### 14.4 命名约定

新增 Query Key 时必须遵守以下规则：

1. **必须使用 Key 工厂对象**：参考 `projectKeys`、`charKeys` 等模式，在对应的 hooks 文件中定义 `xxxKeys` 对象，提供 `all`、`list`、`detail` 等工厂方法。**禁止在 `useQuery` / `invalidateQueries` 中手写裸数组**。
2. **Key 结构统一**：采用 `[实体名, 子类型, ...参数]` 的层级结构。子类型通常为 `"list"` 或 `"detail"`，参数包括 `projectId`、`id` 等。
3. **前缀匹配失效**：使用 `all` 前缀（如 `projectKeys.all = ["projects"]`）可以一次性失效该实体的所有查询。CRUD 操作如果不确定影响范围，优先用 `all` 前缀做全量失效。
4. **列表 Key 绑定 projectId**：列表查询的 key 必须包含 `projectId`，确保切换项目时不会命中旧缓存。
5. **WS 失效用基础前缀**：WS 推送中用 `["tasks"]`、`["assets"]` 等基础前缀做失效，利用 TanStack Query 的前缀匹配特性一次失效所有变体。
6. **新增实体的检查清单**：
   - [ ] 在 hooks 文件中定义 `xxxKeys` 工厂对象
   - [ ] 列表 hook 使用 `xxxKeys.list(projectId)` 作为 queryKey
   - [ ] 详情 hook 使用 `xxxKeys.detail(projectId, id)` 作为 queryKey
   - [ ] Mutation 的 `onSuccess` 中使用 key 工厂对象做 invalidate
   - [ ] 如果任务完成后会影响该实体，在 `useWs.ts` 的 `useTaskWsSubscription` 中添加对应的 `target_type` 分支

## 15. 错误处理设计

> 本项目的错误处理采用**分层捕获**策略，确保任何异常都不会静默丢失，同时给用户和开发者（包括 AI）提供足够的排查信息。

### 15.1 错误分层架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户界面层                                │
│  ┌──────────────┐  ┌──────────────────┐  ┌──────────────────┐  │
│  │    Toast      │  │  ErrorBoundary   │  │  内联错误提示     │  │
│  │ (瞬时通知)    │  │ (渲染崩溃兜底)    │  │ (表单校验等)      │  │
│  └──────┬───────┘  └────────┬─────────┘  └──────────────────┘  │
└─────────┼──────────────────┼───────────────────────────────────┘
          │                  │
          │  toast.error()   │  componentDidCatch
          │                  │  → console.error → logStore
          ▼                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                    logStore（全局日志池）                         │
│  • 最多 500 条，最新的在前                                        │
│  • LogViewer 浮窗实时查看（Ctrl+Shift+L）                         │
│  • 级别过滤：DEBUG / INFO / WARNING / ERROR / CRITICAL            │
│  • AI 排查时的主要信息源                                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │
     ┌─────────────────────────┼───────────────────────────────┐
     │                         │                               │
     ▼                         ▼                               ▼
┌──────────────┐   ┌─────────────────────┐   ┌──────────────────────┐
│ HTTP 错误     │   │ WebSocket 错误       │   │ 浏览器/JS 运行时错误  │
│ (ky 拦截器)   │   │ (WsClient)          │   │ (全局事件监听)        │
├──────────────┤   ├─────────────────────┤   ├──────────────────────┤
│ 4xx→WARNING  │   │ onerror → close()   │   │ window.onerror       │
│ 5xx→ERROR    │   │       → reconnect() │   │   → logStore ERROR   │
│ → logStore   │   │                     │   │                      │
│ → ApiError   │   │ 心跳超时 → close()  │   │ unhandledrejection   │
│ → 业务层      │   │       → reconnect() │   │   → logStore ERROR   │
│   catch/onErr│   │                     │   │                      │
│   → toast    │   │ JSON 解析失败 → 忽略 │   │ console.error 拦截   │
└──────────────┘   └─────────────────────┘   │   → logStore ERROR   │
                                              └──────────────────────┘
```

**数据流向总结**：

- **HTTP 4xx/5xx** → ky `afterResponse` 拦截器 → 写 logStore（4xx=WARNING，5xx=ERROR） → 抛 `ApiError` → 业务层 `onError` / `catch` → `toast.error()`
- **WebSocket 错误** → `WsClient.onerror` → `close()` → 触发 `onclose` → `scheduleReconnect()`（指数退避 + 抖动）
- **React 渲染错误** → `ErrorBoundary.componentDidCatch` → `console.error` → 被 `console.error` 拦截器捕获 → 写 logStore
- **JS 运行时错误** → `window.addEventListener("error")` → 写 logStore（含文件名、行号、堆栈）
- **未处理 Promise rejection** → `window.addEventListener("unhandledrejection")` → 写 logStore（含错误名和堆栈）

### 15.2 错误分类处理表

| 错误类型 | 处理方式 | 用户看到什么 | AI 如何排查 |
| --- | --- | --- | --- |
| **HTTP 4xx**（参数错误、资源不存在等） | ky 拦截器写 logStore（WARNING 级）→ 抛 `ApiError` → hook 的 `onError` 调 `toast.error()` | Toast 弹出错误消息（来自后端 `message` 字段），3 秒自动消失 | logStore 中搜 `WARNING` 级别，查看 `context` 中的 `status`、`url`、`method`、`body` |
| **HTTP 5xx**（后端异常） | ky 拦截器写 logStore（ERROR 级）→ 抛 `ApiError` → hook 的 `onError` 调 `toast.error()` | Toast 弹出错误消息 | logStore 中搜 `ERROR` 级别 + `api` 来源；同时检查后端日志（后端也会记录 5xx） |
| **WebSocket 连接断开** | `WsClient.onclose` → `scheduleReconnect()`（指数退避 1s→30s，±25% 抖动） | 无直接提示（后台静默重连） | logStore 中如果看到 WS 相关消息突然中断，说明连接断了；检查后端是否重启 |
| **WebSocket 心跳超时** | 30 秒发一次 `ping`，服务端回 `pong`；如果长时间无响应，连接会被浏览器关闭 → 触发重连 | 无直接提示 | 检查后端心跳处理是否正常 |
| **WS 推送的 task.failed** | `useTaskWsSubscription` 收到后 invalidate `["tasks"]` + 对应实体缓存 | 任务列表中该任务状态变为"失败"；如果用户在详情页，对应实体的素材会刷新 | 查看任务详情中的 `error_message` 字段；logStore 中搜索对应 `task_id` |
| **WS 推送的 script.failed** | `useScriptWsSubscription` 收到后 invalidate `["script", projectId]` | 剧本页面刷新，显示解析失败状态 | 查看剧本解析状态字段；检查后端 LLM 配置是否正确 |
| **React 渲染异常** | `ErrorBoundary` 捕获 → 显示错误兜底页面 + `console.error` → 被拦截器写入 logStore | 全屏错误页面："页面渲染出错" + 错误消息 + "刷新页面"按钮 | logStore 中搜 `console.error` 来源的 ERROR 日志，包含组件堆栈 |
| **JS 运行时错误**（TypeError、ReferenceError 等） | `window.onerror` 捕获 → 写 logStore（ERROR 级，含文件名、行号、列号、堆栈） | 无直接 UI 提示（但不影响使用的错误会被记录） | logStore 中搜 `window.error` 来源，`context` 包含 `filename`、`lineno`、`colno`、`stack` |
| **未捕获 Promise rejection** | `window.onunhandledrejection` 捕获 → 写 logStore（ERROR 级） | 无直接 UI 提示 | logStore 中搜 `promise` 来源的 ERROR 日志，`context` 含堆栈 |
| **表单校验失败** | 组件内部处理（不经过全局链路） | 表单字段下方内联错误提示 | 无需排查，属于正常业务逻辑 |

### 15.3 设计原则

#### 什么时候用 toast vs ErrorBoundary vs 内联错误

| 场景 | 选择 | 理由 |
| --- | --- | --- |
| **用户主动操作的反馈**（保存、删除、提交） | Toast | 操作是瞬时的，Toast 3 秒自动消失，不打断工作流。所有 mutation 的 `onSuccess` / `onError` 都通过 `toast.success()` / `toast.error()` 反馈 |
| **React 组件渲染崩溃** | ErrorBoundary | 渲染异常如果不捕获会导致整个页面白屏。ErrorBoundary 是最后的防线，显示兜底页面并提供"刷新页面"按钮 |
| **表单字段校验** | 内联错误 | 校验错误是预期内的、局部的，应该在对应字段旁边提示，不需要上升到全局通知 |
| **后台异步事件**（WS 推送的任务失败等） | Toast + 缓存刷新 | 用户可能正在看别的页面，Toast 提供即时通知；同时 invalidate 缓存确保下次切回时数据是最新的 |

#### WebSocket 推送的错误 vs HTTP 返回的错误如何避免重复展示

本项目中，HTTP 和 WS 的错误展示**职责不同，天然不冲突**：

1. **HTTP 请求错误**（如 `POST /api/generate` 返回 500）：这是"提交动作本身失败"，由 mutation 的 `onError` 处理，弹 toast 告诉用户"任务提交失败"。
2. **WS 推送错误**（如 `task.failed`）：这是"任务在后台执行过程中失败"，此时 HTTP 请求早已成功返回（任务已提交）。WS 只是触发缓存刷新，**不会额外弹 toast**。任务失败的状态通过列表/详情页的 UI 呈现（如红色状态标签）。

因此：HTTP 错误 toast 提示的是"操作失败"，WS 推送的是"状态更新"（含失败状态），两者不会重复弹。

**唯一需要注意的边界情况**：`useParseScript` 的 `onSuccess` 会弹 `toast.info("剧本解析已启动")`，后续解析失败时 WS 推送 `script.failed` 只刷新缓存不弹 toast。用户需要在剧本页面看到解析失败的状态——这是正确的设计，因为解析是异步的，失败信息应该嵌在页面数据里而不是 toast。

#### 后台任务（generate）的错误为什么必须走 WS 而不是 HTTP

生成任务（图片/视频）是**异步长时任务**，整个流程是：

```
HTTP POST /api/generate  →  后端创建任务记录，立即返回 202
                              ↓
                          后台 Worker 执行生成（可能耗时 10s ~ 5min）
                              ↓
                          完成/失败 → WS 推送 task.completed / task.failed
                              ↓
                          前端 invalidate ["tasks"] → 列表自动刷新
```

**不能用 HTTP 等待结果的原因**：

1. **超时不可控**：图片生成可能 10 秒，视频生成可能 5 分钟。HTTP 连接不可能保持这么久（网关、代理都可能超时断开）。
2. **批量场景**：`useBatchGenerate` 一次提交多个任务，HTTP 只返回"已提交 N 个"，每个任务的结果必须独立推送。
3. **用户体验**：提交后立即返回"已提交"，用户可以继续操作。任务完成后 WS 即时推送，Toast 通知 + 列表刷新——不需要用户手动刷新。
4. **重试/恢复**：任务状态持久化在数据库中，即使前端刷新页面，`useTasks` 重新拉取也能看到所有任务（包括失败的）。WS 只是加速了这个过程。

**错误排查链路**：如果任务失败了，AI 应该：
1. 查看任务列表中的 `error_message` 字段（通过 API 或数据库）
2. 检查 logStore 中是否有相关的 WS 消息记录
3. 查看后端日志中 Worker 的执行记录
4. 检查 Provider 配置（API Key、模型、参数）是否正确
